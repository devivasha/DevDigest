## Tool & Library Notes

- **2026-07-12** ‚Äî `parseUnifiedDiff` (`server/src/adapters/git/diff-parser.ts`) splits the raw string on `"\n"` and does not flush the current hunk until the next `diff --git` line or end-of-input ‚Äî so if the raw diff string itself ends with a trailing `"\n"`, `String.split("\n")` produces one extra empty `""` element, which the parser silently counts as an unnamed context line and appends one **phantom** extra number to that hunk's `newLineNumbers` (one past the real last line). When handcrafting a unified diff for eval-case seed data (T15 of eval-pipeline), building the diff via `linesArray.join("\n")` (no trailing newline) avoids the phantom line entirely ‚Äî verified by extracting `checkoutDiff`/`uploadDiff` from `seed.ts` and re-running the parser logic standalone: both hunks report exactly the real line count (17 and 18) with no extra entry. If a real git-style diff (which always ends with `\n`) is fed through instead, expect `newLineNumbers` to be one longer than the visible content ‚Äî harmless for grounding (it just marks one extra new-side line as "covered"), but don't build `EvalExpectation.findings[].end_line` off of a hunk's last reported number without checking whether it's real content. ref: `server/src/adapters/git/diff-parser.ts:14` (`parseUnifiedDiff`), `server/src/db/seed.ts` (`checkoutDiff`/`uploadDiff`)

- **2026-07-10** ‚Äî `Buffer#toString('utf8')` never throws on invalid/non-UTF-8 byte sequences ‚Äî it silently substitutes the replacement character `U+FFFD` per Node/WHATWG decoding rules. To detect that a file's bytes weren't valid UTF-8 (needed so `documents.ts`'s `readDocument` can throw instead of silently serving mangled text), round-trip it: decode then re-encode (`Buffer.from(text, 'utf8')`) and `Buffer.compare` against the original buffer ‚Äî a mismatch means the decode step lost information. Cheap for markdown-sized docs; would need chunking for very large files. ref: `server/src/modules/project-context/documents.ts:82`

## Codebase Patterns (fix-loop iteration 2, `docs/plans/export-to-ci.md` ‚Äî PR self-review)

- **2026-07-15** ‚Äî `CiRepository.ingestResults`'s prior "pre-check SELECT, then unconditionally insert `agent_runs`, then `ci_runs` with `ON CONFLICT DO NOTHING`" shape (see the earlier 2026-07-15 insight above documenting it) had a real concurrency gap: two transactions racing on the same `(ci_installation_id, pr_number, ran_at)` key can BOTH pass the pre-check SELECT before either has inserted, so both proceed to insert their own `agent_runs` row even though the `ci_runs` unique constraint only lets one of the two `ci_runs` rows survive ‚Äî `agent_runs` count then exceeds the deduped `ci_runs` count under concurrency, breaking AC-23's 1:1 idempotency. Fixed by reordering: do the `ci_runs` insert with `.onConflictDoNothing({...}).returning()` FIRST for each artifact, then only insert `agent_runs` when `returning()` came back non-empty (i.e. THIS call actually won the constraint). Postgres serializes concurrent `INSERT ... ON CONFLICT DO NOTHING` on the same unique key ‚Äî the loser blocks until the winner commits, then sees the conflict and returns nothing ‚Äî so gating `agent_runs` on a non-empty `returning()` is atomic, unlike gating on a separate pre-check `SELECT`. The pre-check `findExisting` helper is kept but now only runs on the conflict branch (to fetch the winning row's data for the response), never before the insert. ref: `server/src/modules/ci/repository.ts` (`ingestResults`)
