- **2026-07-06** ‚Äî Blast Radius `changed_symbols` only ever contains symbols present in the PERSISTENT `symbols` index (built at a prior indexed SHA). ## Codebase Patterns

- **2026-06-29** ‚Äî Smart Diff classifier (`smart-diff-classifier.ts`) uses priority order `boilerplate ‚Üí wiring ‚Üí core` (first match wins). All pattern arrays and numeric thresholds (`LARGE_PR_THRESHOLD`, `LARGE_FILE_THRESHOLD`) are exported constants ‚Äî never inline in route logic. The route imports these constants so the client can reference thresholds without duplicating them. ref: `server/src/modules/pulls/smart-diff-classifier.ts:1`

- **2026-06-14** ‚Äî Shared contracts (`@devdigest/shared`) are vendored as TWO hand-maintained copies ‚Äî `server/src/vendor/shared/` and `client/src/vendor/shared/` ‚Äî resolved by tsconfig path alias, NOT auto-synced. Adding a field means editing both in lock-step; the only diffs between copies are comments. Evidence: `server/src/vendor/shared/contracts/trace.ts`, `platform.ts`.
- **2026-06-14** ‚Äî PR-list per-PR aggregates (score, cost) are computed ON READ in `GET /repos/:id/pulls` via one `inArray` query + JS grouping, never denormalized onto `pull_requests`. "Latest review batch" cost has no batch id in the schema ‚Äî approximated by summing `agent_runs.cost_usd` within a 120s window of the PR's newest priced run. Evidence: `server/src/modules/pulls/routes.ts`.
- **2026-06-14** ‚Äî `completeAgentRun`'s `values` shape is declared in TWO places that must match: the repo fn (`repository/run.repo.ts`) AND the interface wrapper (`repository.ts:151`). Adding a field (e.g. `costUsd`) needs both or typecheck fails.

- **2026-07-06** ‚Äî The server barrel `vendor/shared/index.ts` uses explicit `.js` extension re-exports (`export * from './contracts/brief.js'`), while the client's mirrored barrel `client/src/vendor/shared/index.ts` omits the extension (`export * from './contracts/brief'`) ‚Äî both compile fine under each package's own `tsconfig`/`moduleResolution`, but a byte-identical copy of the barrel itself would fail one side. Only the per-contract files (e.g. `contracts/blast.ts`) need to be byte-identical; the barrel line must match each side's existing extension convention. ref: `server/src/vendor/shared/index.ts:19`, `client/src/vendor/shared/index.ts:19`

- **2026-07-06** ‚Äî `RepoIntelRepository.getResolvedCallers` (`repository.ts:503`) INNER JOINs `file_rank` on `(repoId, fromPath)`. A `references` row whose `fromPath` has NO matching `file_rank` row is silently dropped from `tryPersistentBlast`'s caller list ‚Äî no error, no degraded flag, the caller just vanishes from the `BlastResponse`. Any seed/backfill of blast demo data must insert a `file_rank` row for every caller file, not just for the changed (decl) file. ref: `server/src/modules/repo-intel/repository.ts:503`

- **2026-07-06** ‚Äî Blast Radius deliberately EXCLUDES test-file callers (and their endpoint/cron fixtures) from the `BlastResponse`. Rationale: a test calling a symbol isn't a production "blast radius", and test files carry endpoint/cron string literals as fixtures ‚Äî on PR #8, `seed`'s 6 callers were all `server/test/*.it.test.ts` and pulled in 20 bogus "impacted endpoints" scraped from those tests. Filter lives in `blast/service.ts` (`isTestFile`, the only consumer of `getBlastRadius`), NOT in the repo-intel facade ‚Äî "what counts as impact" is a feature policy. `isTestFile` matches `*/test|tests|__tests__|__mocks__|e2e/*` path segments and `*.test.*`/`*.spec.*` filenames. Top-level `impacted_endpoints`/`impacted_crons` are recomputed from non-test caller files' `factsByFile` (persistent path); the degraded/ripgrep path has no per-file attribution so it falls back to the unfiltered flat list. ref: `server/src/modules/blast/service.ts:34` (`isTestFile`), `service.ts:88`

- **2026-07-10** ‚Äî T5's `documents.ts` disambiguates "missing file" (should be a 404) from "guard violation" (422) even though `path-guard.ts`'s `assertInsideClone` throws `ValidationError` for BOTH. There is no separate error subclass or code from the guard for "missing" ‚Äî the only signal is the exact message string `"Path does not exist: ${relPath}"` it throws in that specific branch. `readDocument` catches `ValidationError`, checks `err.message.startsWith('Path does not exist:')`, and re-throws as `NotFoundError` only in that case; every other guard message (traversal, absolute path, escape, missing clone root) is left as `ValidationError`. This is a string-prefix contract with `path-guard.ts` ‚Äî if that message text ever changes, this remap silently stops working (typecheck won't catch it). ref: `server/src/modules/project-context/documents.ts:71` (`MISSING_TARGET_PREFIX`), `server/src/modules/project-context/path-guard.ts:85`

- **2026-07-10** ‚Äî T10 wired `run-executor.ts`'s `runOneAgent` to build `specPaths` via `resolveSpecPaths({ agentPaths: agent.attachedDocPaths, loadedSkills })` where `loadedSkills` is `linkedSkills.filter(s => s.skill.enabled).map(s => ({ paths: s.skill.attachedDocPaths }))` ‚Äî the SAME `enabled` filter already applied a few lines above for `skillBodies`, just re-derived rather than reusing a shared variable (kept separate on purpose: skills' prompt bodies and skills' attached-doc paths are conceptually different projections of the same filtered set, and coupling them would make a future "only some enabled skills contribute docs" change harder). No normalization (no `./`-stripping, no trim) is applied before calling `resolveSpecPaths` ‚Äî deliberately, per the resolver's own documented "trusts discovery-canonical input" contract (see the adjacent 2026-07-10 "resolveSpecPaths dedupes by exact string" entry below); normalizing here would be the wrong layer since discovery (T4) is meant to be the sole canonical-path producer. `specTexts`/`readPaths`/`missing` are declared OUTSIDE the per-agent `try` block (not inside) specifically so the `catch` block's failure/cancel trace can report a partial read set (e.g. docs were read successfully but the LLM call afterward threw) instead of always falling back to `[]`. `traceFromBuffer` gained two new optional trailing params (`specsRead`, `specsMissing`, both defaulting to `[]`) so `failAll`'s pre-work-failure call site (before any agent-specific spec resolution has run) is unaffected. ref: `server/src/modules/reviews/run-executor.ts:317` (spec resolution block), `server/src/modules/reviews/run-executor.ts:636` (`traceFromBuffer` signature)

- **2026-07-10** ‚Äî T10 does NOT memoize doc reads across agents in the same `executeRuns` batch ‚Äî each agent re-reads its own `specPaths` from disk via `readDocument` even if two agents attach the identical path. Rationale: attach sets are per-agent (deduped only within one agent's own union by `resolveSpecPaths`), the number of attached docs is expected to be small (a handful of markdown files), and a cross-agent cache would add state/invalidation complexity for a cost (a few extra `fs.readFile` calls) that's negligible next to the per-agent LLM call it feeds. If this ever becomes a real bottleneck (e.g. many agents √ó many large docs), a `Map<path, string>` cache scoped to one `executeRuns` call would be the natural fix ‚Äî reads happen after diff+intent are shared once already, so the precedent exists. ref: `server/src/modules/reviews/run-executor.ts:339` (per-agent read loop)

## Tool & Library Notes

- **2026-07-10** ‚Äî `Buffer#toString('utf8')` never throws on invalid/non-UTF-8 byte sequences ‚Äî it silently substitutes the replacement character `U+FFFD` per Node/WHATWG decoding rules. To detect that a file's bytes weren't valid UTF-8 (needed so `documents.ts`'s `readDocument` can throw instead of silently serving mangled text), round-trip it: decode then re-encode (`Buffer.from(text, 'utf8')`) and `Buffer.compare` against the original buffer ‚Äî a mismatch means the decode step lost information. Cheap for markdown-sized docs; would need chunking for very large files. ref: `server/src/modules/project-context/documents.ts:82`

- **2026-06-14** ‚Äî New DB columns: edit `db/schema/*.ts`, then `npm run db:generate` (drizzle-kit) auto-generates `00NN_*.sql` (e.g. `0010_solid_baron_zemo.sql` = `ALTER TABLE ‚Ä¶ ADD COLUMN`). Never hand-write migration SQL; apply with `npm run db:migrate`.
- **2026-07-06** ‚Äî `db/schema/context.ts`'s `references` table has NO unique index/constraint at all (unlike `symbols`, which has `symbols_repo_path_name_kind_line_uq`). `.onConflictDoNothing()` on an insert into `references` is a no-op guard ‚Äî Postgres has no constraint to conflict on, so re-running an insert unconditionally duplicates rows. Idempotent seeding of `references` requires a manual `SELECT` of existing `(fromPath, toSymbol, line)` keys first, then filtering the insert batch in JS. ref: `server/src/db/seed.ts` (Blast Radius demo data block), `server/src/db/schema/context.ts:97`

- **2026-07-06** ‚Äî For a hermetic unit test of a `class Foo { constructor(private container: Container) {} }` service, don't construct a real `Container` (its constructor builds `LocalNoAuthProvider`/`JobRunner` from a real `db` and can surprise you later if those constructors start doing I/O). Instead build a plain object with only the properties the service actually reads (`{ db, repoIntel, llm }`) and cast `as unknown as Container`. TypeScript's structural typing doesn't check this at the cast site, so it compiles even though most of `Container`'s surface is missing ‚Äî fine as long as the service under test never touches an untyped property (verify by reading the service body first). ref: `server/src/modules/blast/service.test.ts:97`
- **2026-07-06** ‚Äî When a join query is grouped/aggregated in JS after the query returns (e.g. `prFiles` ‚ãà `pullRequests`, then collapsed to one entry per PR with accumulated `files_overlap`), do NOT put `.limit(N)` on the SQL query itself ‚Äî a flat row-level limit can truncate before all of one PR's overlapping-file rows are collected, or cut off entire trailing PRs unevenly. Fetch unbounded (ordered DESC), aggregate in JS, then `.slice(0, N)` on the aggregated list. ref: `server/src/modules/blast/service.ts:150`
