## What Works

- **2026-07-12** — T2 of the eval-pipeline plan (adding `eval_set_runs` NEW table + `eval_runs` += `workspace_id`/`set_run_id`/`agent_version` + a plain `uniqueIndex('eval_cases_owner_name_uq')` on `eval_cases`) generated a SINGLE clean `0013_*.sql` with `pnpm db:generate` and required NO interactive "created or renamed?" prompt — confirms the plan's own prediction that a purely additive (new table + new nullable/backfillable columns on an empty table) schema change never triggers drizzle-kit's rename-detection heuristic, unlike the drop+create pattern documented in the 2026-07-11 onboarding entry below. `eval_runs.workspace_id uuid NOT NULL` with no default applied instantly since the table has zero rows — no rewrite, no backfill needed. ref: `server/src/db/migrations/0013_same_dormammu.sql`

- **2026-07-12** — T5 of the eval-pipeline plan (`EvalRepository`) needed one method beyond the plan's literal method list: `updateSetRunAggregate(workspaceId, setRunId, patch)`. The plan's T5 text only lists `insertSetRun(workspaceId, row): id`, but T7's service narrative later requires "insertSetRun → per-case loop → pool via scorer.poolSetMetrics → **update the set-run aggregate**" — i.e. the set-run row is created BEFORE the per-case loop (so per-case rows have a `set_run_id` to reference) and only gets its recall/precision/citation_accuracy/traces_passed/traces_total/duration/cost/under_min written AFTER all cases are scored. Without an update method, T7 has no way to persist the final aggregate. Added `updateSetRunAggregate` (workspace-scoped update) to close this gap — flag for whoever implements T7 that this is the method to call, not a second `insertSetRun`. ref: `server/src/modules/eval/repository.ts` (`updateSetRunAggregate`)

- **2026-07-12** — T5's `dashboardAllAgents` computes pooled `current`/`delta` in JS by grouping ALL of a workspace's `eval_set_runs` rows (unbounded query, ordered `ranAt desc`) by `${ownerKind}:${ownerId}` and taking each group's first two entries as latest/previous — same "fetch unbounded, aggregate in JS" precedent as `blast/service.ts`'s prior-PRs join (2026-06-14 insight). There is no SQL-native "latest N per group" in the Drizzle query builder used elsewhere in this codebase, and a raw `DISTINCT ON` would only get the single latest row per owner, not the previous one needed for delta — the two-pass JS grouping was simpler than a raw SQL window-function query for what is expected to be a small per-workspace table. ref: `server/src/modules/eval/repository.ts` (`dashboardAllAgents`)

- **2026-07-12** — T5's `EvalRepository` has NO dependency edge on T4 (`scorer.ts`) per the plan's own concurrency waves (both are Wave B, parallel, no cross-import). So the AC-14 regression-alert text ("Precision dipped 2pts") needed for `dashboardForOwner`/`dashboardAllAgents`'s `alert` field is computed by a small **local, duplicate** pure function (`describeRegression` at the bottom of `repository.ts`) rather than importing `scorer.regressionOf`. This is intentional given the DAG, but means T7 (service) may end up with two independent regression-detection implementations (this one for the dashboard aggregate, and `scorer.regressionOf` for whatever T7 wires up) unless T7's author consciously reuses/replaces one with the other. Flag for T7. ref: `server/src/modules/eval/repository.ts` (`describeRegression`)

- **2026-07-12** — T7 (`eval/service.ts`) reconciled the above duplicate by RECOMPUTING `alert` in the service after calling `dashboardForOwner`/`dashboardAllAgents` — `dashboard()`/`dashboardAll()` spread the repo's `EvalDashboard` result and overwrite `alert` with a local `describeAlert()` built on `scorer.regressionOf`, rather than deleting the repo's own `describeRegression` (kept in place per its own doc comment, since removing it would touch T5's `Owned paths`). Net effect: the repo computes a throwaway `alert` that the service always discards/replaces — harmless (cheap pure function) but worth knowing if a future task consolidates the two. ref: `server/src/modules/eval/service.ts` (`describeAlert`, `dashboard`)

- **2026-07-12** — T7's `runSet` treats a case's `input_diff` as "unparseable" (AC-17/Rec-D) using `raw.trim() === '' || parseUnifiedDiff(raw).files.length === 0` — NOT a try/catch on `parseUnifiedDiff` alone, because `diff-parser.ts`'s `parseUnifiedDiff` **never throws** on garbage input; it silently degrades to `{ raw, files: [] }`. A try/catch around it is still present (defensive, matches the plan's literal wording) but is structurally dead code today — the real degraded-detection signal is the empty-files check. Any future test asserting "malformed diff string → degraded" must pass a string that either is blank OR contains no recognizable `diff --git`/`+++`/`@@` markers (a string with some other garbage content but a stray `+++ b/x` line + one `@@` hunk header would still parse to ≥1 file and NOT degrade). ref: `server/src/modules/eval/service.ts` (`tryParseDiff`), `server/src/adapters/git/diff-parser.ts:78`

- **2026-07-12** — T7's `createFromFinding`/`createCase` catch a Postgres unique-violation (`SQLSTATE 23505`, duck-typed via `(err as {code?:string}).code === '23505'`) from `EvalRepository.createCase` and rethrow as a `ValidationError` (422) — there was NO existing precedent for this anywhere in the server codebase (`grep -rn "23505"` across `server/src` returns nothing else), so this is a new pattern. Needed because `EvalRepository.createCase`'s own doc comment explicitly punts this to callers ("callers should treat a unique-violation from Postgres as a validation error, not swallow it here") and `createFromFinding`'s derived case name embeds the finding's own id (`deriveCaseName`), meaning a duplicate/retried "Turn into eval case" click on the SAME finding deterministically collides with the `(workspace_id, owner_id, name)` UNIQUE constraint every time (not a rare race) — an unhandled raw `postgres` driver error would otherwise leak to the client as a 500. ref: `server/src/modules/eval/service.ts` (`isUniqueViolation`, `createCaseGuarded`)

- **2026-07-12** — T8's `eval/routes.ts` builds `CreateEvalCaseBody`/`UpdateEvalCaseBody` as `EvalCaseInput.omit({ owner_kind: true, owner_id: true })` (create) `.partial()` (update) rather than accepting the full `EvalCaseInput` shape in the request body — the plan's T8 text doesn't spell out the exact body schema for the manual-create/update routes, only that `PATCH /eval-cases/:caseId` and `POST /agents/:id/eval-cases` exist. Deriving `owner_kind: 'agent'`/`owner_id: req.params.id` server-side (never trusting a body-supplied owner) mirrors the same IDOR discipline as `createFromFinding` (finding #4) — a client could otherwise create/attribute a case to a different agent than the one in the URL. `service.createCase`/`updateCase` still validate `expected_output` against `EvalExpectation` internally (GAP-4), so the route-level omission doesn't weaken that check. ref: `server/src/modules/eval/routes.ts` (`CreateEvalCaseBody`, `UpdateEvalCaseBody`)

- **2026-07-12** — `EvalService.compare(workspaceId, baseId, headId)` (T7) intentionally does NOT take/cross-check an `agentId` — `GET /agents/:id/eval-compare?base=&head=` still resolves `:id` via `getContext`+`IdParams` for consistency with the sibling nested routes, but the actual tenancy guarantee comes entirely from `EvalRepository.getTwoSetRuns` filtering both `base`/`head` by `workspace_id` (AC-24 is satisfied without an agent-ownership check on this specific endpoint, since a set-run id is already an opaque, workspace-scoped identifier — there's no cross-agent IDOR risk analogous to finding #4 here). ref: `server/src/modules/eval/service.ts` (`compare`), `server/src/modules/eval/routes.ts` (`/agents/:id/eval-compare`)

## What Doesn't Work

- **2026-07-06** — Blast Radius `changed_symbols` only ever contains symbols present in the PERSISTENT `symbols` index (built at a prior indexed SHA). Files CREATED by the PR under review are not in that index, so their symbols never appear in the `BlastResponse` — blast structurally under-reports a feature PR that adds new files. Verified on PR #8 "HW 4 Blast Radius": only pre-existing `OverviewTab`/`page.tsx`/`seed.ts` symbols showed; the new `blast/service.ts`, `blast/routes.ts`, `BlastRadiusCard.tsx` were absent because the index predated them (confirmed with a `SELECT distinct path FROM symbols` returning empty for those paths). Not a bug — a limitation of index-vs-diff mismatch. ref: `server/src/modules/repo-intel/service.ts:315` (`tryPersistentBlast`)

- **2026-07-11** — `MockLLMProvider.completeStructured` (`server/src/adapters/mocks.ts:90`) ALWAYS `safeParse`-validates its `structured`/`structuredBySchema` fixture against the EXACT Zod schema in the request and THROWS if it fails — so it can never itself return a payload violating the schema's own `.max(n)` array-length caps (e.g. onboarding's `criticalPaths.max(7)`, T1's `OnboardingSectionsSchema`). This means a consumer's own post-grounding "belt-and-suspenders" cap-slice logic (e.g. `onboarding/extractor.ts`'s `applyCaps()`, sliced 12→7 items / 1500→1200 char narrative) is UNREACHABLE via `MockLLMProvider` — the fixture would be rejected before ever reaching that code. Testing it required a small custom `LLMProvider` double implementing only `completeStructured` that returns the oversized payload WITHOUT validating it (simulating a real provider/model that doesn't strictly enforce array bounds). ref: `server/src/modules/onboarding/extractor.test.ts` (`makeOvercapProvider`)

- **2026-07-11** — `docs/plans/onboarding-generator.md`'s T12 acceptance text for `extractor.test.ts` says a ">8-node or fenced diagram → null", but tracing `onboarding/extractor.ts`'s `sanitizeDiagram` character-by-character shows a diagram wrapped in markdown code fences (e.g. ` ```mermaid\nflowchart TD\n...\n``` `) is cleanly STRIPPED by its two `.replace(/```.../g, "")` calls and, if the remaining content is a valid ≤8-node flowchart, is NOT nulled — it survives sanitization as-is. Only a genuinely invalid diagram (wrong prefix after stripping, >8 distinct node ids, or an unsafe `click`/`href`/`javascript:` directive) becomes `null`. Not a code bug — LLMs commonly fence mermaid output and stripping+accepting it is the correct UX — but the plan's literal T12 wording should not be read as "any fenced diagram is discarded"; the test file asserts the verified behavior (valid fenced diagram survives; >8-node unfenced diagram and unsafe-directive diagram both become `null`). ref: `server/src/modules/onboarding/extractor.ts:191` (`sanitizeDiagram`)

- **2026-07-11** — `brief/service.ts`'s original `getOrCompute` called `this.repo.getBrief(prId)` (a cache read keyed ONLY by `prId`, no `workspace_id` column on `pr_brief`) BEFORE any tenancy check — a cross-tenant leak on cache hit, since `getBrief`/`upsertBrief` never filter by workspace. `compute()` right below it already did tenancy-first correctly via `getPull(workspaceId, prId)`. General pattern for this codebase: any per-PR cache/read table that lacks its own `workspace_id` column MUST have the caller's tenancy proven via a workspace-scoped `getPull`/`getRepo` lookup BEFORE the cache is touched, even when a sibling method in the same file already "looks" tenancy-safe — each entry point needs its own check, not just the write path. ref: `server/src/modules/brief/service.ts:92` (`getOrCompute`)

- **2026-07-10** — T5's `documents.ts` disambiguates "missing file" (should be a 404) from "guard violation" (422) even though `path-guard.ts`'s `assertInsideClone` throws `ValidationError` for BOTH. This is a string-prefix contract with `path-guard.ts` — if that message text ever changes, this remap silently stops working (typecheck won't catch it). ref: `server/src/modules/project-context/documents.ts:71` (`MISSING_TARGET_PREFIX`), `server/src/modules/project-context/path-guard.ts:85`

- **2026-07-10** — T10 wired `run-executor.ts`'s `runOneAgent` to build `specPaths` via `resolveSpecPaths({ agentPaths: agent.attachedDocPaths, loadedSkills })` where `loadedSkills` is `linkedSkills.filter(s => s.skill.enabled).map(s => ({ paths: s.skill.attachedDocPaths }))` — the SAME `enabled` filter already applied a few lines above for `skillBodies`, just re-derived rather than reusing a shared variable (kept separate on purpose: skills' prompt bodies and skills' attached-doc paths are conceptually different projections of the same filtered set, and coupling them would make a future "only some enabled skills contribute docs" change harder). - **2026-07-10** — T10 does NOT memoize doc reads across agents in the same `executeRuns` batch — each agent re-reads its own `specPaths` from disk via `readDocument` even if two agents attach the identical path. Rationale: attach sets are per-agent (deduped only within one agent's own union by `resolveSpecPaths`), the number of attached docs is expected to be small (a handful of markdown files), and a cross-agent cache would add state/invalidation complexity for a cost (a few extra `fs.readFile` calls) that's negligible next to the per-agent LLM call it feeds. If this ever becomes a real bottleneck (e.g. many agents × many large docs), a `Map<path, string>` cache scoped to one `executeRuns` call would be the natural fix — reads happen after diff+intent are shared once already, so the precedent exists. ref: `server/src/modules/reviews/run-executor.ts:339` (per-agent read loop)

## Tool & Library Notes

- **2026-07-12** — `parseUnifiedDiff` (`server/src/adapters/git/diff-parser.ts`) splits the raw string on `"\n"` and does not flush the current hunk until the next `diff --git` line or end-of-input — so if the raw diff string itself ends with a trailing `"\n"`, `String.split("\n")` produces one extra empty `""` element, which the parser silently counts as an unnamed context line and appends one **phantom** extra number to that hunk's `newLineNumbers` (one past the real last line). When handcrafting a unified diff for eval-case seed data (T15 of eval-pipeline), building the diff via `linesArray.join("\n")` (no trailing newline) avoids the phantom line entirely — verified by extracting `checkoutDiff`/`uploadDiff` from `seed.ts` and re-running the parser logic standalone: both hunks report exactly the real line count (17 and 18) with no extra entry. If a real git-style diff (which always ends with `\n`) is fed through instead, expect `newLineNumbers` to be one longer than the visible content — harmless for grounding (it just marks one extra new-side line as "covered"), but don't build `EvalExpectation.findings[].end_line` off of a hunk's last reported number without checking whether it's real content. ref: `server/src/adapters/git/diff-parser.ts:14` (`parseUnifiedDiff`), `server/src/db/seed.ts` (`checkoutDiff`/`uploadDiff`)

- **2026-07-10** — `Buffer#toString('utf8')` never throws on invalid/non-UTF-8 byte sequences — it silently substitutes the replacement character `U+FFFD` per Node/WHATWG decoding rules. To detect that a file's bytes weren't valid UTF-8 (needed so `documents.ts`'s `readDocument` can throw instead of silently serving mangled text), round-trip it: decode then re-encode (`Buffer.from(text, 'utf8')`) and `Buffer.compare` against the original buffer — a mismatch means the decode step lost information. Cheap for markdown-sized docs; would need chunking for very large files. ref: `server/src/modules/project-context/documents.ts:82`

- **2026-07-11** — `vitest run .it.test` (the CLI positional filename filter used for the integration test job) DOES pick up new `.it.test.ts` files placed under `server/src/modules/**/` — not only files under `server/test/` — confirmed via `vitest list .it.test` and a full `vitest run .it.test` run showing all 9 integration files (including 2 new ones under `src/modules/onboarding/` and `src/modules/repo-intel/`) collected and passing. This matches `vitest.config.ts`'s `include: ['test/**/*.test.ts', 'src/**/*.test.ts']`, under which any `*.it.test.ts` file also satisfies the broader `*.test.ts` glob regardless of directory. (One earlier terminal run's piped/grepped output looked like it excluded the two new files — that was a truncated-log grep artifact, not an actual exclusion; a clean rerun confirmed inclusion.) ref: `server/vitest.config.ts`

- **2026-06-14** — New DB columns: edit `db/schema/*.ts`, then `npm run db:generate` (drizzle-kit) auto-generates `00NN_*.sql` (e.g. `0010_solid_baron_zemo.sql` = `ALTER TABLE … ADD COLUMN`). Never hand-write migration SQL; apply with `npm run db:migrate`.

Idempotent seeding of `references` requires a manual `SELECT` of existing `(fromPath, toSymbol, line)` keys first, then filtering the insert batch in JS. ref: `server/src/db/seed.ts` (Blast Radius demo data block), `server/src/db/schema/context.ts:97`

- **2026-07-06** — For a hermetic unit test of a `class Foo { constructor(private container: Container) {} }` service, don't construct a real `Container` (its constructor builds `LocalNoAuthProvider`/`JobRunner` from a real `db` and can surprise you later if those constructors start doing I/O). Instead build a plain object with only the properties the service actually reads (`{ db, repoIntel, llm }`) and cast `as unknown as Container`. TypeScript's structural typing doesn't check this at the cast site, so it compiles even though most of `Container`'s surface is missing — fine as long as the service under test never touches an untyped property (verify by reading the service body first). ref: `server/src/modules/blast/service.test.ts:97`

Fetch unbounded (ordered DESC), aggregate in JS, then `.slice(0, N)` on the aggregated list. ref: `server/src/modules/blast/service.ts:150`
