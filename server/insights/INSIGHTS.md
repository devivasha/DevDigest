## What Doesn't Work

- **2026-07-06** — Blast Radius `changed_symbols` only ever contains symbols present in the PERSISTENT `symbols` index (built at a prior indexed SHA). Files CREATED by the PR under review are not in that index, so their symbols never appear in the `BlastResponse` — blast structurally under-reports a feature PR that adds new files. Verified on PR #8 "HW 4 Blast Radius": only pre-existing `OverviewTab`/`page.tsx`/`seed.ts` symbols showed; the new `blast/service.ts`, `blast/routes.ts`, `BlastRadiusCard.tsx` were absent because the index predated them (confirmed with a `SELECT distinct path FROM symbols` returning empty for those paths). Not a bug — a limitation of index-vs-diff mismatch. ref: `server/src/modules/repo-intel/service.ts:315` (`tryPersistentBlast`)

- **2026-07-11** — `MockLLMProvider.completeStructured` (`server/src/adapters/mocks.ts:90`) ALWAYS `safeParse`-validates its `structured`/`structuredBySchema` fixture against the EXACT Zod schema in the request and THROWS if it fails — so it can never itself return a payload violating the schema's own `.max(n)` array-length caps (e.g. onboarding's `criticalPaths.max(7)`, T1's `OnboardingSectionsSchema`). This means a consumer's own post-grounding "belt-and-suspenders" cap-slice logic (e.g. `onboarding/extractor.ts`'s `applyCaps()`, sliced 12→7 items / 1500→1200 char narrative) is UNREACHABLE via `MockLLMProvider` — the fixture would be rejected before ever reaching that code. Testing it required a small custom `LLMProvider` double implementing only `completeStructured` that returns the oversized payload WITHOUT validating it (simulating a real provider/model that doesn't strictly enforce array bounds). ref: `server/src/modules/onboarding/extractor.test.ts` (`makeOvercapProvider`)

- **2026-07-11** — `docs/plans/onboarding-generator.md`'s T12 acceptance text for `extractor.test.ts` says a ">8-node or fenced diagram → null", but tracing `onboarding/extractor.ts`'s `sanitizeDiagram` character-by-character shows a diagram wrapped in markdown code fences (e.g. ` ```mermaid\nflowchart TD\n...\n``` `) is cleanly STRIPPED by its two `.replace(/```.../g, "")` calls and, if the remaining content is a valid ≤8-node flowchart, is NOT nulled — it survives sanitization as-is. Only a genuinely invalid diagram (wrong prefix after stripping, >8 distinct node ids, or an unsafe `click`/`href`/`javascript:` directive) becomes `null`. Not a code bug — LLMs commonly fence mermaid output and stripping+accepting it is the correct UX — but the plan's literal T12 wording should not be read as "any fenced diagram is discarded"; the test file asserts the verified behavior (valid fenced diagram survives; >8-node unfenced diagram and unsafe-directive diagram both become `null`). ref: `server/src/modules/onboarding/extractor.ts:191` (`sanitizeDiagram`)

- **2026-07-11** — `brief/service.ts`'s original `getOrCompute` called `this.repo.getBrief(prId)` (a cache read keyed ONLY by `prId`, no `workspace_id` column on `pr_brief`) BEFORE any tenancy check — a cross-tenant leak on cache hit, since `getBrief`/`upsertBrief` never filter by workspace. `compute()` right below it already did tenancy-first correctly via `getPull(workspaceId, prId)`. General pattern for this codebase: any per-PR cache/read table that lacks its own `workspace_id` column MUST have the caller's tenancy proven via a workspace-scoped `getPull`/`getRepo` lookup BEFORE the cache is touched, even when a sibling method in the same file already "looks" tenancy-safe — each entry point needs its own check, not just the write path. ref: `server/src/modules/brief/service.ts:92` (`getOrCompute`)

## What Works

- **2026-06-29** — `deriveFileSummary(patch)` in `smart-diff-classifier.ts` extracts "What this does" text purely from the patch string — no LLM call. Priority: (1) up to 3 exported symbol names → "Adds: rateLimit, bucketKey"; (2) first meaningful added line that isn't an import, comment, or brace; (3) `null` (caller hides the section). ## Codebase Patterns

- **2026-06-29** — Smart Diff classifier (`smart-diff-classifier.ts`) uses priority order `boilerplate → wiring → core` (first match wins). All pattern arrays and numeric thresholds (`LARGE_PR_THRESHOLD`, `LARGE_FILE_THRESHOLD`) are exported constants — never inline in route logic. The route imports these constants so the client can reference thresholds without duplicating them. ref: `server/src/modules/pulls/smart-diff-classifier.ts:1`

- **2026-06-14** — Shared contracts (`@devdigest/shared`) are vendored as TWO hand-maintained copies — `server/src/vendor/shared/` and `client/src/vendor/shared/` — resolved by tsconfig path alias, NOT auto-synced. Adding a field means editing both in lock-step; the only diffs between copies are comments. Evidence: `server/src/vendor/shared/contracts/trace.ts`, `platform.ts`.
- **2026-06-14** — PR-list per-PR aggregates (score, cost) are computed ON READ in `GET /repos/:id/pulls` via one `inArray` query + JS grouping, never denormalized onto `pull_requests`. "Latest review batch" cost has no batch id in the schema — approximated by summing `agent_runs.cost_usd` within a 120s window of the PR's newest priced run. Evidence: `server/src/modules/pulls/routes.ts`.
- **2026-06-14** — `completeAgentRun`'s `values` shape is declared in TWO places that must match: the repo fn (`repository/run.repo.ts`) AND the interface wrapper (`repository.ts:151`). Adding a field (e.g. `costUsd`) needs both or typecheck fails.

- **2026-07-06** — The server barrel `vendor/shared/index.ts` uses explicit `.js` extension re-exports (`export * from './contracts/brief.js'`), while the client's mirrored barrel `client/src/vendor/shared/index.ts` omits the extension (`export * from './contracts/brief'`) — both compile fine under each package's own `tsconfig`/`moduleResolution`, but a byte-identical copy of the barrel itself would fail one side. Only the per-contract files (e.g. `contracts/blast.ts`) need to be byte-identical; the barrel line must match each side's existing extension convention. ref: `server/src/vendor/shared/index.ts:19`, `client/src/vendor/shared/index.ts:19`

- **2026-07-06** — `RepoIntelRepository.getResolvedCallers` (`repository.ts:503`) INNER JOINs `file_rank` on `(repoId, fromPath)`. A `references` row whose `fromPath` has NO matching `file_rank` row is silently dropped from `tryPersistentBlast`'s caller list — no error, no degraded flag, the caller just vanishes from the `BlastResponse`. Any seed/backfill of blast demo data must insert a `file_rank` row for every caller file, not just for the changed (decl) file. ref: `server/src/modules/repo-intel/repository.ts:503`

- **2026-07-06** — Blast Radius deliberately EXCLUDES test-file callers (and their endpoint/cron fixtures) from the `BlastResponse`. Rationale: a test calling a symbol isn't a production "blast radius", and test files carry endpoint/cron string literals as fixtures — on PR #8, `seed`'s 6 callers were all `server/test/*.it.test.ts` and pulled in 20 bogus "impacted endpoints" scraped from those tests. Filter lives in `blast/service.ts` (`isTestFile`, the only consumer of `getBlastRadius`), NOT in the repo-intel facade — "what counts as impact" is a feature policy. `isTestFile` matches `*/test|tests|__tests__|__mocks__|e2e/*` path segments and `*.test.*`/`*.spec.*` filenames. Top-level `impacted_endpoints`/`impacted_crons` are recomputed from non-test caller files' `factsByFile` (persistent path); the degraded/ripgrep path has no per-file attribution so it falls back to the unfiltered flat list. ref: `server/src/modules/blast/service.ts:34` (`isTestFile`), `service.ts:88`

- **2026-07-10** — T5's `documents.ts` disambiguates "missing file" (should be a 404) from "guard violation" (422) even though `path-guard.ts`'s `assertInsideClone` throws `ValidationError` for BOTH. There is no separate error subclass or code from the guard for "missing" — the only signal is the exact message string `"Path does not exist: ${relPath}"` it throws in that specific branch. `readDocument` catches `ValidationError`, checks `err.message.startsWith('Path does not exist:')`, and re-throws as `NotFoundError` only in that case; every other guard message (traversal, absolute path, escape, missing clone root) is left as `ValidationError`. This is a string-prefix contract with `path-guard.ts` — if that message text ever changes, this remap silently stops working (typecheck won't catch it). ref: `server/src/modules/project-context/documents.ts:71` (`MISSING_TARGET_PREFIX`), `server/src/modules/project-context/path-guard.ts:85`

- **2026-07-10** — T10 wired `run-executor.ts`'s `runOneAgent` to build `specPaths` via `resolveSpecPaths({ agentPaths: agent.attachedDocPaths, loadedSkills })` where `loadedSkills` is `linkedSkills.filter(s => s.skill.enabled).map(s => ({ paths: s.skill.attachedDocPaths }))` — the SAME `enabled` filter already applied a few lines above for `skillBodies`, just re-derived rather than reusing a shared variable (kept separate on purpose: skills' prompt bodies and skills' attached-doc paths are conceptually different projections of the same filtered set, and coupling them would make a future "only some enabled skills contribute docs" change harder). No normalization (no `./`-stripping, no trim) is applied before calling `resolveSpecPaths` — deliberately, per the resolver's own documented "trusts discovery-canonical input" contract (see the adjacent 2026-07-10 "resolveSpecPaths dedupes by exact string" entry below); normalizing here would be the wrong layer since discovery (T4) is meant to be the sole canonical-path producer. `specTexts`/`readPaths`/`missing` are declared OUTSIDE the per-agent `try` block (not inside) specifically so the `catch` block's failure/cancel trace can report a partial read set (e.g. docs were read successfully but the LLM call afterward threw) instead of always falling back to `[]`. `traceFromBuffer` gained two new optional trailing params (`specsRead`, `specsMissing`, both defaulting to `[]`) so `failAll`'s pre-work-failure call site (before any agent-specific spec resolution has run) is unaffected. ref: `server/src/modules/reviews/run-executor.ts:317` (spec resolution block), `server/src/modules/reviews/run-executor.ts:636` (`traceFromBuffer` signature)

- **2026-07-10** — T10 does NOT memoize doc reads across agents in the same `executeRuns` batch — each agent re-reads its own `specPaths` from disk via `readDocument` even if two agents attach the identical path. Rationale: attach sets are per-agent (deduped only within one agent's own union by `resolveSpecPaths`), the number of attached docs is expected to be small (a handful of markdown files), and a cross-agent cache would add state/invalidation complexity for a cost (a few extra `fs.readFile` calls) that's negligible next to the per-agent LLM call it feeds. If this ever becomes a real bottleneck (e.g. many agents × many large docs), a `Map<path, string>` cache scoped to one `executeRuns` call would be the natural fix — reads happen after diff+intent are shared once already, so the precedent exists. ref: `server/src/modules/reviews/run-executor.ts:339` (per-agent read loop)

## Tool & Library Notes

- **2026-07-10** — `Buffer#toString('utf8')` never throws on invalid/non-UTF-8 byte sequences — it silently substitutes the replacement character `U+FFFD` per Node/WHATWG decoding rules. To detect that a file's bytes weren't valid UTF-8 (needed so `documents.ts`'s `readDocument` can throw instead of silently serving mangled text), round-trip it: decode then re-encode (`Buffer.from(text, 'utf8')`) and `Buffer.compare` against the original buffer — a mismatch means the decode step lost information. Cheap for markdown-sized docs; would need chunking for very large files. ref: `server/src/modules/project-context/documents.ts:82`

- **2026-07-11** — `vitest run .it.test` (the CLI positional filename filter used for the integration test job) DOES pick up new `.it.test.ts` files placed under `server/src/modules/**/` — not only files under `server/test/` — confirmed via `vitest list .it.test` and a full `vitest run .it.test` run showing all 9 integration files (including 2 new ones under `src/modules/onboarding/` and `src/modules/repo-intel/`) collected and passing. This matches `vitest.config.ts`'s `include: ['test/**/*.test.ts', 'src/**/*.test.ts']`, under which any `*.it.test.ts` file also satisfies the broader `*.test.ts` glob regardless of directory. (One earlier terminal run's piped/grepped output looked like it excluded the two new files — that was a truncated-log grep artifact, not an actual exclusion; a clean rerun confirmed inclusion.) ref: `server/vitest.config.ts`

- **2026-06-14** — New DB columns: edit `db/schema/*.ts`, then `npm run db:generate` (drizzle-kit) auto-generates `00NN_*.sql` (e.g. `0010_solid_baron_zemo.sql` = `ALTER TABLE … ADD COLUMN`). Never hand-write migration SQL; apply with `npm run db:migrate`.
- **2026-07-06** — `db/schema/context.ts`'s `references` table has NO unique index/constraint at all (unlike `symbols`, which has `symbols_repo_path_name_kind_line_uq`). `.onConflictDoNothing()` on an insert into `references` is a no-op guard — Postgres has no constraint to conflict on, so re-running an insert unconditionally duplicates rows. Idempotent seeding of `references` requires a manual `SELECT` of existing `(fromPath, toSymbol, line)` keys first, then filtering the insert batch in JS. ref: `server/src/db/seed.ts` (Blast Radius demo data block), `server/src/db/schema/context.ts:97`

- **2026-07-06** — For a hermetic unit test of a `class Foo { constructor(private container: Container) {} }` service, don't construct a real `Container` (its constructor builds `LocalNoAuthProvider`/`JobRunner` from a real `db` and can surprise you later if those constructors start doing I/O). Instead build a plain object with only the properties the service actually reads (`{ db, repoIntel, llm }`) and cast `as unknown as Container`. TypeScript's structural typing doesn't check this at the cast site, so it compiles even though most of `Container`'s surface is missing — fine as long as the service under test never touches an untyped property (verify by reading the service body first). ref: `server/src/modules/blast/service.test.ts:97`
- **2026-07-06** — When a join query is grouped/aggregated in JS after the query returns (e.g. `prFiles` ⋈ `pullRequests`, then collapsed to one entry per PR with accumulated `files_overlap`), do NOT put `.limit(N)` on the SQL query itself — a flat row-level limit can truncate before all of one PR's overlapping-file rows are collected, or cut off entire trailing PRs unevenly. Fetch unbounded (ordered DESC), aggregate in JS, then `.slice(0, N)` on the aggregated list. ref: `server/src/modules/blast/service.ts:150`

- **2026-07-11** — T2 of the onboarding-generator plan: `pnpm db:generate` prompts an interactive arrow-key menu ("Is X table created or renamed from another table?") whenever a migration both drops one table and creates a similarly-named one in the same run (here: dropping `onboarding`, creating `onboarding_tours`). - **2026-07-11** — The divergent-lineage trap in `gotchas.md` ("`db:migrate` silently no-ops if the DB was migrated on another branch") does NOT block newly-generated migrations from applying — it only makes `db:migrate` skip migrations whose `when` timestamp is OLDER than the latest applied `when`. On this session's local `devdigest-postgres` container, `drizzle.__drizzle_migrations` had 3 rows (ids 10-12) from a foreign branch lineage sitting between local `0008` and `0011` (hashes didn't match any local `.sql` file), yet `pnpm db:migrate` still correctly applied this session's freshly-generated `0012_funny_unicorn.sql` as a new row (id 14) because its `when` timestamp (generated just now) is newer than all existing rows. The no-op trap only bites when the LOCAL new migration's timestamp happens to be older than what's already recorded as applied — freshly generated migrations are always newest, so they're safe. Verified via `psql \d onboarding_tours` (table + FKs + both indexes present) and `\d onboarding` (relation not found) after migrate. ref: `server/insights/gotchas.md` ("db:migrate silently no-ops...")
